## Project Overview

### Apps
- **central** (`apps/central`) â€” Admin management interface (Vite SPA, port 3001)
- **notices** (`apps/notices`) â€” Notices/meetings/insecticides management (Vite SPA, port 3002)

### Packages

- **lib** (`packages/lib`) â€” Business logic, constants, validation schemas (Zod)
- **supabase** (`packages/supabase`) â€” Supabase client, auth functions, database collections, generated types
- **typescript-config** (`packages/typescript-config`) â€” Shared TS configs (base, react-library, tanstack-start)
- **assets** (`packages/assets`) â€” Static images synced to app public directories

## Common Commands

```bash
# Install dependencies
pnpm install

# Run all dev servers
pnpm dev

# Run a single app dev server
pnpm --filter central dev
pnpm --filter notices dev
pnpm --filter public dev

# Build all apps
pnpm build

# Build a single app
pnpm turbo run build --filter=central

# Type checking
pnpm check-types

# Lint (Biome)
pnpm lint

# Run tests (supabase package)
pnpm --filter @mcmec/supabase test        # watch mode
pnpm --filter @mcmec/supabase test:run     # single run

# Changesets (versioning)
pnpm changeset
pnpm version-pkgs
```

- **TanStack Query** for server state, **TanStack Form** for form management, **TanStack Table** for data tables
- **Supabase** backend (PostgreSQL + Auth); local dev via `supabase/config.toml` (API port 54321, DB port 54322, Studio port 54323)

- **Biome** for linting and formatting (no ESLint/Prettier)
- **Vite 7** for bundling; **Nitro** for SSR server in public app
- **Vitest** for unit tests
- **Zod v4** for schema validation

## Architecture Notes

- Apps import shared packages via workspace protocol (`@mcmec/ui`, `@mcmec/lib`, `@mcmec/supabase`)
- `central` and `notices` are client-side SPAs; `public` is SSR with TanStack Start outputting to `.output/public/`
- Each app syncs shared assets before building (`pnpm sync-assets` copies from `@mcmec/assets`)
- Route trees are auto-generated by TanStack Router plugin (`routeTree.gen.ts` â€” do not edit)
- Database types are auto-generated in `packages/supabase/src/database.types.ts` â€” do not edit manually
- The `packages/ui/src/components/` directory contains generated shadcn-style components excluded from Biome linting

## Code Style & Formatting

- **Biome** is the sole linter/formatter â€” configured at repo root `biome.json`
- Tab indentation, double quotes for JS/TS

- Import organization enforced automatically

## Accessibility

The public app (`apps/public`) must be **WCAG 2.1 AA** compliant. When working on this app or on shared UI components used by it:
- Use semantic HTML elements (`nav`, `main`, `article`, `section`, etc.)
- Ensure all interactive elements are keyboard accessible
- Provide meaningful `alt` text for images and `aria-label`/`aria-labelledby` for non-text controls
- Maintain minimum color contrast ratios (4.5:1 for normal text, 3:1 for large text)
- Ensure form inputs have associated `<label>` elements
- Support screen readers â€” test with proper heading hierarchy and landmark regions

## Development Workflow

All changes go through branches and pull requests â€” never commit directly to `main` or `develop`.

### Branching model
- **`main`** â€” production. Vercel deploys and Supabase migrations apply on merge.
- **`develop`** â€” staging. All feature branches merge here first.
- **Feature branches** â€” created from `develop` (e.g., `feat/contact-form`, `fix/auth-redirect`, `chore/update-deps`)

### Feature â†’ develop â†’ main flow
1. **Create a feature branch** from `develop`
2. **Make changes and commit** â€” pre-commit hooks auto-run Biome lint/format on staged files
3. **Add a changeset** if the PR affects app behavior: `pnpm change` â€” skip for CI/config-only changes
4. **Push and open a PR to `develop`** â€” the PR template pre-fills a checklist; auto-labeler tags the PR by affected area
5. **CI runs automatically** â€” lint, type-check, build, and tests must all pass
6. **Review, resolve conversations, and squash merge** into `develop`
7. **When ready to release**, open a PR from `develop` â†’ `main`, review the combined diff, and merge
8. **Vercel deploys only affected apps** to production on merge to `main`

### Preview deployments
Vercel preview deploys are **off by default** on all branches (including `develop`). To trigger one, include `[deploy-preview]` in a commit message.

### Database changes
- Supabase migrations in `supabase/migrations/` **auto-apply to the production database** when merged to `main` (not `develop`)
- Always test migrations locally first: `supabase db reset` (applies migrations + seed data)
- Seed data for local dev is in `supabase/seeds/001_seed.sql`
- Generate updated types after schema changes: `pnpm gen-types`

### CI checks on every PR
- **Lint, Types & Build** â€” `pnpm biome lint`, `pnpm turbo run check-types`, `pnpm turbo run build`
- **Tests** â€” `pnpm --filter @mcmec/supabase test:run`
- **Changeset check** â€” warns (non-blocking) if no changeset is included
- CI runs on PRs to both `develop` and `main`

### Branch protection
**`main`** (strict):
- PRs required â€” no direct pushes
- CI status checks must pass before merge
- Branches must be up to date before merge
- Force pushes blocked

**`develop`** (relaxed):
- PRs required â€” no direct pushes
- CI status checks must pass before merge
- Force pushes allowed
- Branches do not need to be up to date
